# CLAUDE.md

## 0. 项目定位

本项目是一个 CAD / SolidWorks / STEP 零部件自动匹配与装配关系识别项目。

输入是一批 CAD 零件文件。它们之间存在真实的、唯一确定的匹配关系。匹配组可能是：

- 1 个零件单独成组；
- 2 个零件一组；
- 3 个零件一组；
- 4 个或更多零件一组。

典型例子：

- 杯子、杯盖、杯套可能构成一个三零件匹配组；
- 两个能互相装配的盘、盖、套、壳体等应被识别为同一组；
- 不同型号的杯子和杯盖不能混配；
- 不属于同一真实匹配关系的零件，即使几何上局部贴合，也不应被错误装配到一起。

本项目不是普通图像识别项目，也不是简单的几何相似度检索项目。核心问题是：

> 在 CAD 零件池中，找出物理上能装、工程语义上合理、并且唯一对应的零部件匹配组。

当前阶段优先目标不是论文包装，而是形成可靠、可调试、可复现实验的工程闭环。

默认使用中文回答。

英文专业名词第一次出现时，需要给出中文解释，例如：

- beam search（束搜索）
- collision validation（碰撞验证）
- pose residual（位姿残差）
- constraint graph（约束图）
- backtracking（回溯）
- Boolean Common volume（布尔交集体积）

回答风格应保持严肃、工程化、批判性。

不要使用夸张表达，例如：

- “重大突破”
- “完美解决”
- “肯定能发论文”
- “这个一定是创新点”

如果证据不足，应明确说明：

- 当前只能做工程判断；
- 该结论尚未验证；
- 需要哪些实验才能确认。

## 2. Claude Code 的角色

你是本项目中的执行型代码助手，不是论文概念生成器。

你的主要职责是：

1. 快速读懂项目结构；
2. 找出关键问题；
3. 提出最小可验证修改；
4. 编写可运行代码；
5. 运行或指导运行实验；
6. 保持日志和报告清晰；
7. 帮助用户判断阶段性结果是否有效；
8. 在必要时重新审视前面的判断，检查是否发生范围膨胀或幻觉。

你不应该：

1. 主动扩展大方向；
2. 为了显得高级而引入复杂框架；
3. 在没有证据时声称算法有效；
4. 用 Agent / RL / LLM 包装本来可以用几何规则解决的问题；
5. 忽略 CAD 几何验证；
6. 忽略多零件组合的物理合理性；
7. 自动把工程问题升级成论文系统设计问题。

用户偏好：

- 直接；
- 严肃；
- 工程闭环优先；
- 先指出问题，再谈方案；
- 不接受空泛创新点；

- 希望每一步都有可验证结果；

- 用户自己负责判断路线是否正确。

2. 依据；

4. 具体命令；

- 一次性列十几个方向；
- 没有实验支撑的论文式包装；
- 没有证据的乐观判断。

## 4. 当前工程目标

当前阶段优先目标是跑通一个可靠、可调试、可复现实验的工程原型。

优先级从高到低是：

1. 几何验证可靠；
2. 匹配结果可解释；
3. 错误原因可追踪；
4. 多零件组不要退化成“简单贴合堆砌”；
5. 输出 accepted / review / rejected / unresolved 等清晰状态；
6. 保留日志，便于人工检查和下一轮修正；
7. 能稳定复现实验结果；
8. 再考虑算法优化和系统化整理。

不要优先追求：

- 强化学习；
- 端到端深度学习；
- 图像装配；
- 复杂神经网络；
- 大规模重构；
- 没有评价指标的 Agent 包装；
- 没有实验支撑的论文创新包装。

第一阶段核心思想：

> 几何负责“能不能装”，规则和语义负责“该不该装”，Agent 负责调度、诊断、重试和报告。

## 6. 已知技术方向和风险点

项目中可能已经涉及以下技术：

- complete-pose backtracking（完整位姿回溯）
- beam search（束搜索）
- connected next-part choices（连通候选零件选择）
- OCCT collision validation（OCCT 碰撞验证）
- Boolean Common volume（布尔交集体积）
- selected-constraint residual（选中约束残差）
- rejection reason logging（拒绝原因记录）
- score-ranked complete poses（按分数排序的完整位姿）
- accepted / review / rejected / unresolved 状态输出

需要特别警惕的问题：

如果求解器只保留最高分完整位姿，那么最高分位姿一旦被精确碰撞检测拒绝，系统可能无法回退到低分但无碰撞的候选。这会导致：

- 明明存在可行位姿，却被误判失败；
- 碰撞检测变成最终否决器，而不是候选筛选器；
- 搜索过程不可解释；
- 日志中看不到低分候选是否可行。

处理这类问题时，优先检查：

1. 是否保留 complete beam；
2. 是否按分数顺序检查多个完整位姿；
3. 是否记录每个候选的 rank；
4. 是否记录每个拒绝原因；
5. 是否先筛残差，再做精确碰撞；
6. 是否存在过早剪枝；
7. 是否存在只优化 score、不验证物理可行性的情况。

不要一上来更换整体算法。

## 7. 算法判断原则

几何“贴上了”不等于真实“装配成立”。

判断一个匹配关系是否可信，应综合考虑：

1. 接触面或连接特征是否合理；
2. 轴线、孔、槽、边界是否存在对应关系；
3. 位姿残差是否足够小；
4. 是否发生实体穿透；
5. 是否只是局部碰巧贴合；
6. 多零件组合是否形成稳定、现实的装配关系；
7. 是否存在同类零件混配风险；
8. 是否有更合理的候选组；
9. 结果是否可被日志解释；
10. 人工复核时是否能看懂为什么接受或拒绝。

对于 3/4-part 及以上组合，尤其要防止：

- 简单堆叠；
- 表面贴片式拼接；
- 没有功能关系的几何吸附；
- 单个强匹配零件把弱相关零件拖进组；
- 只优化分数，不验证物理可行性；
- 为了通过测试而放宽规则。

## 8. 输出状态定义

最终结果不应只输出“成功/失败”。建议使用以下状态。

### accepted

高置信接受。通常要求：

- 几何约束成立；
- 碰撞检测通过；
- 位姿残差可接受；
- 组合结构合理；
- 没有明显更优冲突候选；
- 日志能够解释接受原因。

### review

需要人工复核。通常包括：

- 几何上可能成立，但工程语义不够明确；
- 多个候选分数接近；
- 组合形态可疑；
- 局部接触成立但整体装配关系不强；
- 数据不足以直接 accepted。

### rejected

明确拒绝。通常包括：

- 碰撞穿透；
- 残差过大；
- 关键连接特征不对应；
- 组合结构明显不合理；
- 与已接受匹配冲突；
- 文件解析失败导致结果不可用。

### unresolved

暂不能判断。通常包括：

- 缺少必要几何特征；
- CAD 文件解析失败；
- 候选不足；
- 当前规则无法覆盖；
- 需要补充实验或人工标注。

## 9. 长时间工作模式

本项目允许 Claude Code 进行较长时间的连续工作，而不是每一步都限制在 1-2 个文件内。

但连续工作必须以“阶段”为单位推进。每个阶段开始前，先说明：

1. 本阶段目标；
2. 计划检查或修改的文件范围；
3. 为什么这些文件属于同一阶段；
4. 本阶段不处理哪些问题；
5. 成功判据是什么；
6. 可能风险是什么；
7. 是否需要用户确认。

阶段内可以修改多个文件，但必须保持任务边界清晰。不要因为发现相关问题就无限扩展范围。

阶段规模建议：

- 小阶段：1-2 个文件，适合 bug 修复；
- 中阶段：3-6 个文件，适合一个完整功能链路；
- 大阶段：超过 6 个文件，必须先说明必要性，并拆成清晰子任务。

如果工作超过一个明显阶段，或者修改超过 5 个文件，必须主动暂停并输出阶段报告。
